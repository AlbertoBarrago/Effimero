Architecture

A small, boring, single-responsibility codebase. pnpm workspaces; TypeScript everywhere except the static HTML packages.

Monorepo layout

effimero/
  apps/
    server/       Fastify API: ingest, stats, registry, swagger, static serving
    dashboard/    React cockpit dashboard (Vite)
    website/      static presentation site
  packages/
    snippet/      browser tracking script (~700 B, esbuild IIFE)
  examples/
    test-site/    plain HTML pages for local hit generation
  docs/           source documentation
  Dockerfile      multi-stage build: server + snippet + dashboard in one image

Data flow

browser ──POST /collect──▶ Fastify
                             │  getSiteConfig(redis)        registry.ts
                             │  isOriginAllowed(origin)     origins.ts
                             │  getDailySalt(redis)         salt.ts
                             │  visitorHash(ip, ua, salt)   hash.ts
                             │  deriveDimensions(ip, ua)    enrichment.ts
                             ▼
                           recordHit(redis)                 stats.ts
                             │  PFADD uniques + live, INCR pageviews,
                             │  HINCRBY paths/referrers/dimensions/hours
                             ▼
dashboard ◀─GET /stats,/live─ getStats / getLiveVisitors  (scoped by auth.ts)

A hit is recorded only when the siteId is registered and the request Origin passes the site's allow-list. Reads are gated by auth.ts, which resolves the bearer credential to a scope (all sites for the admin key, one site for a read token).

Each module has a single reason to change, which keeps the SOLID story honest: index.ts knows HTTP, stats.ts knows Redis, and neither knows the other's internals.

Key design decisions

Registry-gated, origin-checked ingest. /collect records a hit only when the siteId is registered and the request Origin is within the site's allow-list (empty = any). Both checks are a single HGETALL and fail closed with a silent 204, so the endpoint neither injects nor leaks anything for unknown sites.

Two-level read access. The global STATS_API_KEY is the admin credential (registry management + read all). Each site also gets a read token whose SHA-256 is stored behind a reverse index; presenting it grants read access to that one site only. Scope resolution in auth.ts is a pure function, unit-tested without HTTP.

Server-side hashing. The browser does not know its public IP, so the hash cannot be computed client-side. The server computes it in memory per request and discards the inputs. The payload carries no identity at all.

Redis HyperLogLog for uniques. Counting uniques exactly would require storing every hash (a de-anonymization surface and unbounded memory). HLL gives ~0.81% error in at most 12 KB per key and makes the stored data non-reversible even in theory.

Salt via SET NX + TTL instead of a cron job. Any request can create the day's salt; Redis atomicity guarantees a single winner under concurrency. No scheduler, no clock-skew bugs, and replicas agree by construction.

Daily keys with per-key TTL instead of a cleanup job. Retention falls out of Redis expiry; there is nothing to garbage-collect.

Fastify with JSON Schema on every route. Validation happens before handlers run, and the same schemas generate the OpenAPI document at /docs/api. One source of truth.

One Docker image. The multi-stage build compiles snippet, dashboard, and server, and the server serves the static assets. Self-hosters deploy exactly two containers.

Performance characteristics

  • /collect does one config lookup, one salt lookup, and one pipelined Redis transaction (about 12 commands, single round trip).
  • /stats reads the whole range in one pipeline; a 90-day query is roughly 900 Redis commands in one round trip, still single-digit milliseconds on a local Redis.
  • The server is stateless: horizontal scaling is adding replicas behind the proxy.
