Getting started

1. Start the server

git clone https://github.com/albertobarrago/effimero && cd effimero
docker compose up -d
curl http://localhost:3000/health   # {"status":"ok","redis":true}

2. Add the snippet

<script src="https://your-host/effimero.js" data-site="my-site" defer></script>
AttributeRequiredDescription
data-siteyesSite identifier. Letters, digits, . _ -, max 64 chars.
data-endpointnoCollect URL. Defaults to /collect on the snippet's host.

The snippet weighs about 700 bytes, tracks SPA navigations (pushState and popstate), and disables itself when the browser sends Do Not Track or Global Privacy Control.

3. Unlock the dashboard

Read endpoints are protected by STATS_API_KEY. If you did not set one, Effimero generates a key on startup:

docker compose logs effimero | grep generated

Open the dashboard at http://localhost:3000/ or https://your-host/, paste the key when prompted, and enter your site id. Auto-generated keys change whenever the server restarts; set STATS_API_KEY in Compose to keep access stable.

4. Watch the data

The dashboard shows live visitors, daily uniques and pageviews, top pages and referrers, browsers, OS, devices, languages, countries, and pageviews by hour.

Self-hosting & configuration

Effimero is one Docker image plus Redis. All configuration is environment variables:

VariableDefaultDescription
PORT3000HTTP listen port
REDIS_URLredis://localhost:6379Redis connection string
ALLOWED_ORIGINS*Comma-separated CORS origins allowed to POST hits. Restrict in production.
TRUST_PROXYfalseTrust X-Forwarded-For. Required behind any reverse proxy.
RETENTION_DAYS90Days of aggregate stats kept in Redis
STATS_API_KEYauto-generatedBearer key protecting /stats, /live, and /sites. Unset: a random key is generated and logged once at boot (docker compose logs effimero | grep generated). Set an explicit value to keep dashboard access stable across restarts. Set disabled to make reads public.

Dashboard access key

The dashboard uses the same bearer key as API reads. Retrieve the generated key with:

docker compose logs effimero | grep generated

For production, set a stable value and rotate it by changing the environment variable and restarting the effimero service.

services:
  effimero:
    environment:
      STATS_API_KEY: "change-me"

The one setting that bites: behind nginx, Caddy, or Traefik you must set TRUST_PROXY=true, otherwise every visitor appears to come from the proxy IP and unique counts flatline at 1.

nginx example

location / {
    proxy_pass http://effimero:3000;
    proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    proxy_set_header Host $host;
}

Hardening

  • Restrict ALLOWED_ORIGINS to the sites embedding the snippet.
  • Keep Redis inside the Docker network.
  • Keep STATS_API_KEY set to a strong value. Optionally add proxy auth for the dashboard, while leaving /collect and /effimero.js public.
  • Multiple server replicas are safe: salt creation is atomic (SET NX), the server is stateless.

API reference

The interactive, always-current reference is the Swagger UI served by the API at /docs/api (OpenAPI JSON at /docs/api/json). Summary:

Authentication: read endpoints require Authorization: Bearer <STATS_API_KEY>. Without configuration a random key is generated at every boot and printed once in the server logs; the dashboard asks for it and keeps it in sessionStorage for the tab's lifetime. /collect stays public by design: it is written to by your visitors' browsers and carries no identity.

POST /collect

{ "siteId": "my-site", "path": "/pricing", "referrer": "https://google.com/" }

Records one pageview. Returns 204, or 400 on schema violation. The payload carries no identity: the visitor hash is computed server-side from connection IP and User-Agent, in memory only.

GET /stats/{siteId}?range={days}

Aggregates for the last range days (default 30): days[] with uniques and pageviews, totals, topPaths, topReferrers, hours[24], and breakdowns for browsers, os, devices, languages, countries.

Accuracy: daily uniques are HyperLogLog counts (~0.81% error). The range-wide total merges daily-salted HLLs, so a returning visitor counts once per day: treat it as an upper bound.

GET /live/{siteId}

Unique visitors in roughly the last 5 minutes: { "live": 7 }

GET /health

{ "status": "ok", "redis": true }, degraded when Redis is unreachable.

Privacy model

Every pageview computes, in memory on the server:

visitorHash = SHA-256(IP | User-Agent | dailySalt | siteId)
  • IP: used for the hash and a GeoIP country lookup, then discarded. Never logged or stored.
  • User-Agent: hashed, and parsed into coarse buckets (browser family, OS, device class). Raw string never stored.
  • dailySalt: 32 random bytes per UTC day, held only in Redis with a TTL. Not derivable from the date. Gone at midnight.
  • siteId: isolates sites, so the same visitor on two Effimero sites yields unrelated hashes.

Without the daily salt, an IP+UA hash would be a persistent pseudonymous identifier, correlatable across days and reversible via rainbow tables over the IPv4 space. The rotation breaks both: yesterday's hashes share no relationship with today's, and reversing requires a salt that no longer exists. Even the operator cannot reconstruct a visitor's history.

What is stored, exactly

Per site and UTC day: one HyperLogLog sketch (individual hashes not recoverable), one pageview counter, and counter hashes for paths, referrers, browsers, OS, devices, languages, countries, hours. There is no event log. No row corresponds to one visitor or one visit.

Deliberately impossible

  • Returning visitors and retention curves: identity does not survive midnight UTC.
  • Cross-site tracking: the siteId in the hash isolates sites.
  • Funnels, journeys, session replay: no per-visitor data exists.
  • Reconstructing an individual's activity, even with full database access.

Consent

Effimero processes the IP transiently, stores no personal data, and cannot single out an individual. Under the prevailing interpretation (matching CNIL guidance on exempt audience measurement) this does not require prior consent, but regulations evolve: verify with your own counsel. What Effimero guarantees is technical: there is no stored personal data to disclose, export, or delete.

Architecture

browser ──POST /collect──▶ Fastify
                             │  getDailySalt(redis)        salt.ts
                             │  visitorHash(ip, ua, salt)  hash.ts
                             │  deriveDimensions(ip, ua)   enrichment.ts
                             ▼
                           recordHit(redis)                stats.ts
                             │  PFADD uniques + live, INCR pageviews,
                             │  HINCRBY paths/referrers/dimensions/hours
                             ▼
dashboard ◀──GET /stats,/live── getStats / getLiveVisitors

Each server module has a single reason to change: index.ts knows HTTP, stats.ts knows Redis, enrichment.ts knows parsing, and none knows the others' internals. JSON Schemas in schemas.ts both validate requests and generate the OpenAPI document: one source of truth.

Key decisions

  • Server-side hashing: the browser does not know its public IP, and this way the payload carries no identity at all.
  • HyperLogLog: exact counting would mean storing every hash (a de-anonymization surface). HLL gives ~0.81% error in at most 12 KB per key, and is non-reversible by construction.
  • Salt via SET NX + TTL: no cron, no clock skew, replicas agree atomically.
  • Per-key TTL retention: cleanup falls out of Redis expiry, nothing to garbage-collect.
